Prelab: BLE Logging + Data Pipeline

Command protocol

I used the course BLE string protocol (cmd:value with delimiter |) to (1) start a timed run and (2) stream a CSV log back to the laptop. This makes tuning repeatable without a USB cable.

Python (Notebook) commands

The notebook triggers runs and pulls back the log from RX_STRING. 

# Start PID run: setpoint_mm | run_ms | Kp | Ki
ble.send_command(CMD.START_PID, "304|50000|0.3|0.002")
print(ble.receive_string(ble.uuid['RX_STRING']))  # "STARTED"

# Download log (CSV lines between PID_BEGIN and PID_END)
ble.send_command(CMD.SEND_PID_LOG, "")

Hardware Setup

ToF placement

A single VL53L1X ToF sensor is mounted at the front of the robot, facing forward to measure distance to the wall. (Only one sensor was used to avoid I2C address conflicts and simplify control.)

Motor driver notes

Motors are driven using PWM output (analogWrite()) to a motor driver. Direction is selected by swapping which input receives PWM (the opposite input is set to 0). A small motor deadband exists, so low commands may not move the robot.

Controller Implementation

ToF readout (non-blocking)

The ToF measurement is implemented as a non-blocking state machine (start ranging → check ready → read distance), so the main loop does not stall waiting for the sensor.

PID / PI control loop

Error is computed as err = setpoint_mm - tof_mm. The motor command is computed from proportional and integral terms (PI). Output is clamped to motor limits. 

// PI controller (conceptual)
err = setpoint_mm - tof_mm;
integ += err * dt;
u = clamp(Kp*err + Ki*integ, -255, 255);
set_motor_cmd(u);

(5000) Integrator wind-up protection

When the motor command saturates or the robot cannot move due to deadband/friction, the integral term can continue accumulating (wind-up), leading to overshoot and oscillation. I added wind-up protection using conditional integration and integrator clamping.

    Parameter tuning

    I tuned Kp and Ki to balance response speed and stability. Larger Kp increases responsiveness but can cause oscillations; Ki reduces steady-state error but can lead to wind-up if not protected.

    • Final setpoint: 304 mm
    • Final gains: Kp = 0.3, Ki = 0.002
    • Motor deadband compensation: MOTOR_DEAD_PWM = 40 and U_DEADBAND = 10

    Extrapolation (Decouple ToF Rate and PID Loop Rate)

    The ToF sensor produces new measurements at a limited rate. If the PID controller only updates when a new ToF sample arrives, the control loop runs at the same slow rate. In this section, I decoupled the PID loop from the ToF update rate: the PID computation runs every loop iteration, while the ToF measurement is updated only when new data is ready.

    1) ToF update rate (sensor rate)

    I measured the ToF update rate by recording a timestamp only when checkForDataReady() returned true and a new distance value was accepted. The average (or median) update period was:

    • ToF update period: XX ms
    • ToF update rate: YY Hz

    2) Decoupled control loop (PID runs every loop)

    The main loop runs the PID controller every iteration. If no new ToF sample is ready, the controller uses the last saved distance value (baseline). This makes the PID loop rate faster than the ToF sensor rate.

    Key idea: update the measurement only when new ToF data arrives, but compute PID every loop.

    // Pseudocode
if (tof_new_ready) {
  d_meas = tof_mm;       // update the measurement used by PID
}
u = PID(d_used);         // compute PID every loop (even if no new ToF)
set_motor_cmd(u);

    3) PID loop rate (controller rate) vs ToF rate

    I measured the PID loop rate by counting PID iterations in a 1-second window (or by logging PID timestamps). The results show the PID loop runs significantly faster than the ToF update rate:

    • PID loop rate: ZZ Hz
    • ToF rate: YY Hz

    4) Linear extrapolation using the last two ToF readings

    Instead of reusing an old datapoint when no new ToF sample is ready, I implemented a simple linear extrapolation using the last two ToF readings. Let the most recent two valid ToF samples be (t1, d1) and (t2, d2) (in seconds and mm). The slope is:

    slope = (d2 - d1) / (t2 - t1)

    For any time t_now after t2 (until the next real ToF update arrives), the estimated distance is:

    d_est(t_now) = d2 + slope * (t_now - t2)

    The PID controller uses d_est when no new ToF reading is available. When a new ToF reading arrives, the two-point history is updated and a new slope is computed.

    Key code snippets

    (A) Update ToF and maintain the last two samples. We only update the history when new ToF data is ready. 

    // Global state (minimal)
volatile bool tof_new = false;
float d_meas = 0.0f;        // latest accepted ToF (mm)
float d1 = 0.0f, d2 = 0.0f; // last two ToF distances (mm)
float t1 = 0.0f, t2 = 0.0f; // last two ToF times (s)

// Call in loop (non-blocking ToF)
void update_tof_nonblocking() {
  if (!tof_ok) return;

  if (distanceSensor.checkForDataReady()) {
    int d = distanceSensor.getDistance();
    distanceSensor.clearInterrupt();
    distanceSensor.stopRanging();

    // Accept only reasonable values (example guard)
    if (d > 50 && d < 4000) {
      d_meas = (float)d;
      tof_new = true;

      // shift history
      t1 = t2;  d1 = d2;
      t2 = millis() / 1000.0f;
      d2 = d_meas;
    }
  }
}

    (B) Extrapolate when no new ToF sample is available. 

    float estimate_distance(float t_now_s) {
  // If we do not have two valid samples yet, fall back to measured distance
  if (t2 <= t1) return d_meas;

  float slope = (d2 - d1) / (t2 - t1);          // mm/s
  float d_est = d2 + slope * (t_now_s - t2);    // mm

  // Optional clamp for safety
  if (d_est < 0) d_est = 0;
  if (d_est > 6000) d_est = 6000;
  return d_est;
}
    // In update_pid(), every loop:
float t_now = millis() / 1000.0f;

// Baseline: use last measured value
// float d_used = d_meas;

// Extrapolation: use estimated distance when no new sample
float d_used = tof_new ? d_meas : estimate_distance(t_now);
tof_new = false;

float err = (float)setpoint_mm - d_used;
integ += err * dt;
int u = clamp(Kp * err + Ki * integ, -255, 255);
set_motor_cmd(u);

    Discussion

    Main observations:

    • ToF updates are slower than the CPU loop; rate-limited logging and non-blocking ranging prevent BLE/data bottlenecks.
    • Motor deadband and quantization can cause the robot to stall near the setpoint unless gains/minimum PWM are chosen carefully.
    • Wind-up protection improves stability when commands saturate or when the robot cannot move at low control outputs.
